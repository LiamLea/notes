### 概述

#### 1.ACME (Automated Certificate Management Environment)

A protocol (RFC 8555) for automating certificate issuance and renewal between a CA and a client (e.g. cert-manager).

Flow: client proves domain ownership via a **Challenge** → CA issues an **Order** (the certificate).

url: `https://acme-v02.api.letsencrypt.org/directory`

##### (1) letsencrypt

- free, automated, open CA run by ISRG
- implements ACME server — cert-manager acts as the ACME client
- rate limits apply on production (50 certs/domain/week)

#### 2.privateKeySecretRef

- stores the ACME **account private key** used to authenticate with Let's Encrypt
- auto-generated by cert-manager if the Secret doesn't exist (triggers account registration)

#### 3.challenge solvers

Cert-manager looks at the `spec.solvers` list from top to bottom (or by specificity). It uses Selectors to decide which one to pick.

##### (1) http01

- proves domain ownership by serving a token at `http://<domain>/.well-known/acme-challenge/<TOKEN>`
- cert-manager creates a **temporary** Ingress rule pointing to a solver pod that serves the token
- Let's Encrypt sends an HTTP GET to that URL; if the token matches, the challenge passes
- **requires**: domain must be publicly reachable via HTTP
- **does not support**: wildcard certificates

```yaml
solvers:
- http01:
    ingress:
      class: nginx
```

##### (2) dns01

- proves domain ownership by creating a TXT record `_acme-challenge.<domain>` with a token derived from the ACME account key
- cert-manager uses the DNS provider's API to create/delete the record automatically
- Let's Encrypt queries DNS for the TXT record; if found, the challenge passes
- **supports**: wildcard certificates (`*.example.com`)
- **requires**: DNS provider API credentials (e.g. Route53, Cloudflare)

```yaml
solvers:
- dns01:
    route53:
      region: ap-northeast-1
      hostedZoneID: ZXXXXXXXXXXXXX
      role: <iam_role>
      secretAccessKeySecretRef:
        name: ""

# or use secrets or node instance profile to autheticate
```

##### (3) TLS-ALPN-01

- The "Temporary" Cert: Cert-manager generates a tiny, temporary self-signed certificate.

- The Contents: This cert is special. It doesn't look like a normal web cert; it contains the specific ACME challenge token inside the "Subject Alternative Name" (SAN) extension.

- The Validation: Let's Encrypt connects, sees this temporary cert, reads the token, and says "Yep, you own the domain."

- Replacement: Once validated, this temporary cert is immediately deleted and replaced by the real Let's Encrypt cert.

#### 4.workflow (Certificate -> CertificateRequest -> Order -> Challenge)

##### (1) certificate

- declares the desired certificate
  - domains
  - issuer
  - secret name
  - duration

##### (2) CertificateRequest

- a one-shot request for a signed certificate; contains a CSR (Certificate Signing Request)

##### (3) Order (订单)

- the Order is the specific record of your conversation with the Certificate Authority (CA) like Let's Encrypt.
- it holds the "menu" of chanllenge options provided by letsencrypt

```yaml
spec:
  dnsNames:
  - mad-biz-console.stg.mad.qorcommerce.com
status:
  authorizations:
  - challenges:
    - token: 04myF9QhNgCcKxLG1EW1mmvVcCJO--hKiTmMNkb5F5M
      type: http-01
      url: https://acme-v02.api.letsencrypt.org/acme/chall/xx/xx/Yl00bw
    - token: 04myF9QhNgCcKxLG1EW1mmvVcCJO--hKiTmMNkb5F5M
      type: tls-alpn-01
      url: https://acme-v02.api.letsencrypt.org/acme/chall/xx/xx/u5FBrQ
    - token: 04myF9QhNgCcKxLG1EW1mmvVcCJO--hKiTmMNkb5F5M
      type: dns-01
      url: https://acme-v02.api.letsencrypt.org/acme/chall/xx/xx/rD-SQg
```

##### (4) Challenge

- tell how to do the challenge (http01 or dns)
- certmanager will do the challenge and then perform a self-check
- Once the self-check passes, certmanager tells letsencrypt to validate

```yaml
spec:
  authorizationURL: https://acme-v02.api.letsencrypt.org/acme/authz/xx/xx
  dnsName: www.aiops.com
  issuerRef:
    group: cert-manager.io
    kind: ClusterIssuer
    name: letsencrypt
  solver:
    http01:
      ingress:
        class: nginx
  # choose one solver
  type: HTTP-01
  token: 04myF9QhNgCcKxLG1EW1mmvVcCJO--hKiTmMNkb5F5M

# http://www.aiops.com/.well-known/acme-challenge/04myF9QhNgCcKxLG1EW1mmvVcCJO--hKiTmMNkb5F5M

# if dns01: _acme-challenge.www.aiops.com 04myF9QhNgCcKxLG1EW1mmvVcCJO--hKiTmMNkb5F5M
```

#### 5.certs

##### (1) validity & renewal

- Let’s Encrypt certificates are valid for **90 days**
- cert-manager renews them at **~2/3 of lifetime** (default: 60 days after issuance, i.e. 30 days before expiry)
- controlled by `Certificate.spec.renewBefore` (default `720h`) and `spec.duration` (default `2160h`)
- cert-manager watches expiry and triggers renewal automatically; no manual intervention needed

##### (2) check all public certs

* https://crt.sh
* search: `%.aiops-dev.com`

### 使用
#### 1.创建证书签发机构
##### （1）方式一：Issuer（指定命名空间中的证书签发机构）
Issuer只能在Issuer所在命名空间中签发证书

```shell
#ca.crt和ca.key需要提前生成
kubectl create secret tls ca-key-pair \
   --cert=ca.crt \
   --key=ca.key \
   --namespace=default
```
```yaml
#issuer.yaml
apiVersion: cert-manager.io/v1alpha3
kind: Issuer
metadata:
  name: ca-issuer
  namespace: default
spec:
  ca:
    secretName: ca-key-pair
```

##### （2）方式二：ClusterIssuer（全局证书签发机构）

* 使用自己的证书
```yaml
apiVersion: cert-manager.io/v1alpha3
kind: ClusterIssuer
metadata:
  name: ca-issuer
spec:
  ca:
    #这个secret要放在cert manager controller命名空间中
    secretName: ca-key-pair
```

* 使用第三方证书
注意：
如果利用这种方式，生成证书，证书可能一直处于waiting状态（即没有申请到）
可能的原因是当前所在的域达到了申请次数
```yaml
#issuer.yaml
apiVersion: cert-manager.io/v1alpha3
kind: ClusterIssuer
metadata:
  name: xx
spec:
  acme:
    #是 acme 协议的服务端，我们这里用 Let's Encrypt，这个地址就写死成这样就行
    server: https://acme-v02.api.letsencrypt.org/directory

    #如果证书过期了会发邮件（过期会自动重新生成）
    email: xx@qq.com

    privateKeySecretRef:
      #指示此签发机构的私钥将要存储到哪个 Secret 对象中
      name: xx    

    solvers:
    - http01:
        ingress:
          class: nginx
```

注意：
  * letsencrypt通过acme协议自动申请证书，其中包含两个sloving Challenge: http01,dns01,主要用来证明域名是属于你所有。
    * http01的校验原理是给你域名指向的 HTTP 服务增加一个临时 location ，`Let’s Encrypt` 会发送 http 请求到 `http:///.well-known/acme-challenge/`，`YOUR_DOMAIN` 就是被校验的域名，`TOKEN`是 ACME 协议的客户端负责放置的文件，在这里 ACME 客户端就是 cert-manager，它通过修改 Ingress 规则来增加这个临时校验路径并指向提供 `TOKEN` 的服务。此方法仅适用于给使用 Ingress 暴露流量的服务颁发证书，并且不支持泛域名证书。
    * dns01 的校验原理是利用 DNS 提供商的 API Key 拿到你的 DNS 控制权限， 在 Let’s Encrypt 为 ACME 客户端提供令牌后，ACME 客户端 (cert-manager) 将创建从该令牌和您的帐户密钥派生的 TXT 记录，并将该记录放在 `_acme-challenge.`。 然后 Let’s Encrypt 将向 DNS 系统查询该记录，如果找到匹配项，就可以颁发证书。此方法不需要你的服务使用 Ingress，并且支持泛域名证书。


#### 2.生成证书(Certificate)
本质就是**Secret**资源
##### （1）方式一：手动生成
```yaml
#cert.yaml
apiVersion: cert-manager.io/v1alpha2
kind: Certificate
metadata:
  name: xx
  namespace: xx
spec:
  secretName: xx      #将生成的证书保存在xx这个secret中
  issuerRef:
    name: xx          #证书签发机构（issuer）的名字
    kind: Issuer      #还可以设为ClusterIssuer
  commonName: <FQDN>  #域名
  organization:
  - CA
  dnsNames:
  - <FQDN>            #还可以用于哪些域名
  - <FQDN>
```

##### （2）方式二：创建Ingress时，自动生成
```yaml
metadata:
  annotations:

    #使用clusterissuer
    cert-manager.io/cluster-issuer: xx

    #或者使用issuer
    cert-manager.io/issuer: xx
spec:
  rules: ...
  tls:
  - hosts:
    #在证书中设置的主机名
    - xx

    #相关证书都存放在secret中，指定secret
    #如果上面设置了颁发机构，则这里会自动生成该secret，不需要提前生成
    secretName: xx
```
